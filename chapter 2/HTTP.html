HTTP/1.1: Keep-Alive & HOL Blocking

HTTP/1.1 introduced persistent connections (Keep-Alive), reusing a single TCP connection for multiple requests, dramatically reducing overhead. However, it created a new problem: Head-of-Line (HOL) Blocking. Requests are processed sequentially, so a slow request (like a large file) blocks all requests behind it, even if they are small and fast. This version's text-based nature was also vulnerable to request smuggling.

HTTP/2: Multiplexing

HTTP/2 solves HOL blocking by introducing streams. Multiple requests and responses can be sent in parallel over a single TCP connection, with data packets interleaved. A slow response no longer blocks others. It also uses a binary protocol, which is more efficient and eliminates the text-parsing ambiguities that caused request smuggling in HTTP/1.1.
Problem: It's still all on one TCP connection. If a single TCP packet is lost, *all* streams must wait for it to be re-sent (TCP-level HOL blocking).

HTTP/3: QUIC over UDP

HTTP/3 replaces TCP with QUIC, a new transport protocol built on UDP. QUIC re-implements streams and reliability *independently*. If a packet for one stream is lost, it does not block any other streams. This finally eliminates all forms of HOL blocking. QUIC also features faster handshakes (0-RTT) for quicker connection startup.
